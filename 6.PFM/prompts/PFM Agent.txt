# 역할
너는 Python 기반 B2C 핀테크 데이터 분석가이자, LLM과 데이터베이스를 결합하여 자동 금융 리포트를 빌드하는 AI 에이전트 웹 솔루션 엔지니어다.
이미 통합 완료된 상반기 가계부 데이터베이스를 기반으로, 평소에는 대화형 채팅창으로 동작하여 사용자의 질문을 SQL로 자동 변환해 분석하고, 사용자가 원할 때만 고품질 DOCX 보고서를 다운로드하는 "Streamlit 기반 AI 자산 관리 및 PFM SQL 에이전트 시스템(app.py)" 소스코드를 작성하라.

# UI 헤더 및 타이틀 규칙 (필수)
Streamlit 애플리케이션의 메인 헤더 및 페이지 타이틀은 반드시 "개인 맞춤형 자산 관리 가계부 Agent"로 지정하여 출력하라.

# 핵심 구조 (두 가지 모드의 명확한 분리)
[기본 모드 - 대화형 SQL Agent]: 자연어 질문 입력 -> LLM 기반 SQL 변환 및 실행 -> Pandas 분석 -> Plotly 인터랙티브 시각화 -> 해석 및 잔소리 브리핑 (빠르고 가볍게 구동)

[리포트 모드 - 보고서 생성]: 사용자가 버튼을 눌렀을 때만 실행 -> 종합 소비 통계 산출 + 외부 트렌드 검색 + 정적 차트 이미지 백업 -> python-docx 기반 종합 가계부 진단 리포트 파일 생성 및 다운로드 제공 (무겁고 정밀하게 구동)

# 프로젝트 구조 및 경로 (고정)
프로젝트 루트: ./6.PFM/

data/user_card_history_merged.csv  → 이미 1월~5월 내역이 병합 완료되어 있는 읽기 전용 마스터 CSV 파일

code/app.py        → 지금 네가 새로 작성해야 하는 Streamlit 메인 애플리케이션 소스코드

output/charts/       → 보고서용 정적 차트 PNG 이미지 파일 백업 저장 경로

output/reports/      → 최종 docx 결과 보고서 파일 저장 경로

# 데이터베이스 테이블 정보 (가계부 마스터 스펙)
대상 테이블 또는 연동 데이터프레임 구조: user_card_history

컬럼 구조: month, Approved_Num, Approved_DateTime, Card_Name, Store_Name, Category, Amount_KRW

주요 품목/가맹점 (Store_Name) 예시: 배달의민족, 쿠팡이츠, 스타벅스, 메가커피, 카카오T(택시), 지하철기본요금, 시내버스, 올리브영, 쿠팡, 무신사, 다이소, 일반편의점, 넷플릭스, 네이버플러스 멤버십 등
※ 프로그램 구동 시 로컬 SQLite(user_card_history.db) 또는 pandas 데이터프레임을 바인딩하여 사용자의 자연어 지시를 SQL 쿼리(또는 Pandas Query)로 매핑해 데이터를 분석하라.

# UI 및 에이전트 요구사항
1. 대화형 분석 영역 (기본 채팅 UI)
st.chat_message와 st.chat_input을 활용하여 자산 관리 챗봇 화면을 빌드하라. (입력창 안내: "나의 소비 내역이나 가계부에 대해 질문하세요.")

사용자가 질문을 던지면 내부적으로 LLM이 적절한 SQL 쿼리를 생성 및 검증하여 실행하고, 결과를 화면에 출력하라.

사용자의 질문 의도(추이 묻기, 비중 비교 등)에 따라 Plotly 기반 인터랙티브 차트(꺾은선, 막대, 파이 차트 등)를 대화창 내에 동적으로 즉각 렌더링하라.

2. 리포트 생성 영역 (선택 실행)
에이전트 분석 답변 하단에 "위 분석 내용과 시각화 그래프를 기반으로 정식 자산 관리 보고서(DOCX)를 생성해드릴까요?"라는 안내 메시지와 리포트 생성 버튼을 노출하라.

사용자가 버튼을 클릭하면 현재까지의 대화 컨텍스트, 상반기 소비 요약, OpenAI 실시간 웹 검색 기반 금융 트렌드 정보를 종합 수집하여 DOCX 파일 생성을 시작하라.

생성이 완료되면 화면에 다운로드 버튼을 활성화하고, output/reports/ 경로에 자동 저장하라. (★ 사용자가 버튼을 누르기 전에는 절대로 미리 DOCX 파일을 생성하지 말 것)

3. 사이드바 제어 패널
LLM 모델 선택 컴포넌트(예: gpt-4o, gpt-4-turbo 등)를 명확히 구현하라.

OpenAI 실시간 금융 트렌드 웹 검색(web_search) 온오프 체크박스를 제공하라. (구현 권장: openai SDK 기반의 OpenAI().responses.create(model="gpt-5-nano", tools=[{"type": "web_search"}], ...) 구조 연동 대비, API 호출 방어 로직 설계)

대화 내역 및 캐시를 완전히 초기화하는 '대화 초기화(Reset Chat)' 버튼을 구현하라.

# 기능별 핵심 마이그레이션 규칙
A. 품목 및 카테고리 추출 엔진
사용자의 질문이나 생성된 SQL WHERE 조건절에서 어떤 카테고리('식비/배달', '카페/간식' 등) 또는 특정 가맹점('배달의민족', '택시' 등)에 관한 질문인지 식별하는 문자열 직접 매칭 및 LLM 추출 로직을 구축하라.

B. 그래프 섹션 및 DOCX 리포트 생성 규칙
기본 화면용 인터랙티브 차트는 Plotly를 사용하되, 리포트 생성 버튼 작동 시에는 무거운 외부 의존성인 kaleido를 절대 사용하지 마라.

리포트 생성을 위해 데이터프레임의 통계를 기반으로 Matplotlib 또는 Seaborn을 호출하여 정적 PNG 이미지 차트를 다시 생성한 뒤, output/charts/에 저장하고 doc.add_picture() 메서드를 통해 DOCX 문서 내부에 직접 시각화 그림으로 삽입하는 방식을 취하라.

보고서 문서는 표지, 요약 문단, 세부 소비 데이터 분석, 이미지 그래프 섹션, 외부 트렌드 원인 분석, 종합 재무 결론 순서로 짜임새 있게 빌드하라.

C. 대화 응답 포맷 제한 규칙 (매우 중요)
챗봇 채팅창 및 리포트에 들어갈 텍스트 답변은 가독성과 서식 꼬임 방지를 위해 마크다운 기호 사용을 철저히 금지하거나 최소화하라. (줄바꿈과 순수 텍스트 위주로 출력할 것)

타이틀 및 대분류 제목에 #, ## 등 마크다운 태그를 쓰지 말고 일반 문단 텍스트로 표기하라.

굵은 글씨(), 인라인 코드(\``), 블록인용문(>`)을 출력하지 마라.

데이터 목록이나 수치 요약이 필요할 때는 순수 문자열 기반의 불렛 리스트(예: "- 식비 지출: 99만 원", "- 교통 지출: 16만 원") 형태로만 출력하라.

D. 환경 변수 로드 및 API Key 방어 규칙
프로젝트 환경 변수 파일(.env)의 절대경로 기준(fintech/.env 또는 지정된 글로벌 프로젝트 경로)으로 load_dotenv()를 명시적으로 수행하라.

OpenAI API Key가 누락되었거나 통신 오류(401, 할당량 부족 등) 발생 시 프로그램이 멈추거나 크래시(Crash) 나지 않도록 st.error 경고창을 띄우고 작동을 안전하게 제한하는 예외 처리(try-except) 코드를 촘촘하게 작성하라.
